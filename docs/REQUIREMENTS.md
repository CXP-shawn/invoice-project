# 发票管理系统 - 需求与技术栈

## 一、需求罗列

### 1. 功能需求

| 模块 | 需求描述 | 分值 |
|------|----------|------|
| **上传** | 支持图片格式的发票文件上传 | 10分 |
| **识别** | 提取发票中的关键要素，并将要素与原始发票图片一同存储 | 30分 |
| **查询** | ① 按日期范围检索<br>② 支持文字模糊查找<br>③ 点击单条发票展示对应发票图片 | 20分 |
| **统计** | ① 按日期范围 + 文字（模糊查找）进行统计<br>② 显示：发票总张数、总金额 | 20分 |

### 2. 技术要求

| 类别 | 要求 |
|------|------|
| **数据库** | MySQL 5.6 及以上版本 |
| **数据库工具** | Navicat（开发/管理用） |
| **程序类型** | 桌面程序 或 Web 程序 |
| **编程语言** | 不限制 |

### 3. 发票关键要素（识别需提取）

- 发票代码、发票号码
- 开票日期
- 购方/销方名称、税号
- 金额、税额、价税合计
- 商品/服务名称
- 其他 OCR 可识别字段

---

## 二、技术栈方案

### 方案 A：Web 应用（推荐）

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **前端** | React + Ant Design / Vue3 + Element Plus | 表单、表格、图片预览、日期范围选择 |
| **后端** | Node.js (Express/Koa) 或 Python (FastAPI/Flask) | RESTful API，文件上传、业务逻辑 |
| **OCR 识别** | PaddleOCR / 百度智能云 OCR / 腾讯云 OCR | 发票关键要素提取（中文发票支持好） |
| **数据库** | MySQL 5.6+ | 存储发票元数据、识别结果 |
| **文件存储** | 本地磁盘 / MinIO / 云存储 | 存储原始发票图片 |
| **开发工具** | Navicat | 数据库设计、调试、数据管理 |

### 方案 B：桌面应用

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| **框架** | Electron + React/Vue | 跨平台桌面应用 |
| **后端逻辑** | 同上，内嵌或本地服务 | 业务逻辑可复用 |
| **OCR / 数据库** | 同方案 A | |

---

## 三、推荐技术栈（Web 方案）

```
┌─────────────────────────────────────────────────────────────┐
│                        前端 (Frontend)                        │
│  React 18 + TypeScript + Ant Design + React Query             │
│  - 上传组件、表格、日期选择、模糊搜索、图片预览                 │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                        后端 (Backend)                         │
│  Node.js + Express + TypeScript  或  Python + FastAPI          │
│  - 文件上传、OCR 调用、CRUD、统计接口                          │
└─────────────────────────────────────────────────────────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│   MySQL 5.6+    │ │  PaddleOCR /     │ │  本地/云存储     │
│   发票元数据     │ │  百度/腾讯 OCR   │ │  发票图片文件    │
└─────────────────┘ └─────────────────┘ └─────────────────┘
```

### 技术选型理由

| 技术 | 理由 |
|------|------|
| **React + Ant Design** | 组件丰富，表格、上传、日期选择、模态框开箱即用 |
| **Node.js 或 Python** | Node 生态成熟；Python 在 OCR、数据处理上更顺手，可按熟悉度选择 |
| **PaddleOCR** | 开源、免费、中文发票识别效果好，可离线部署 |
| **MySQL** | 满足题目要求，Navicat 可直接连接管理 |

---

## 四、数据库设计要点

- `invoices` 表：发票元数据（识别出的字段 + 图片存储路径）
- 索引：开票日期、金额、关键文字字段，便于按日期和模糊查询
- 图片路径：本地相对路径或 URL，便于点击展示
